# Memory System Reimplementation Plan

## Overview

Replace the current flat key-value memory system with a searchable, linked knowledge base. The system must support:

1. **Project namespacing** — memories belong to named projects/topics (e.g. "locolm", "user-preferences")
2. **Keyword search** — FTS5 full-text search across all memories
3. **Linked memories** — explicit typed relationships between memories (the graph/triplet model)
4. **Automatic recall** — load high-priority memories at conversation start
5. **On-demand search** — find memories by keywords or traverse links during conversation

## Design Principles

1. **SQLite only** — no new dependencies. modernc.org/sqlite provides FTS5.
2. **Simple structures** — tabular with JSON columns. No external graph DB.
3. **LLM-driven retrieval** — the LLM decides when to search, what keywords to use, which memories are relevant.
4. **Two-tier loading** — automatic recall of high-priority items, plus on-demand search/link traversal.
5. **Minimal tool surface** — nine tools: `memory_recall`, `memory_search`, `memory_save`, `memory_get`, `memory_forget`, `memory_link`, `memory_unlink`, `memory_links`, `memory_stats`.

## Database Schema

### Schema version

```sql
PRAGMA user_version = 1;
```

The `user_version` pragma tracks the current schema version. On startup, `memory.go` reads it and runs migration SQL if the version is behind. Future schema changes increment this value and add migration steps:

```go
func migrate(db *sql.DB) {
    var version int
    db.QueryRow("PRAGMA user_version").Scan(&version)

    if version < 1 {
        // Initial schema creation (see tables below)
        db.Exec("PRAGMA user_version = 1")
    }
    if version < 2 {
        // Future migration example:
        // db.Exec("ALTER TABLE memories ADD COLUMN new_col TEXT")
        db.Exec("PRAGMA user_version = 2")
    }
}
```

### memories table

```sql
CREATE TABLE IF NOT EXISTS memories (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    key TEXT UNIQUE NOT NULL,
    value TEXT NOT NULL,
    project TEXT NOT NULL DEFAULT 'general',
    priority INTEGER NOT NULL DEFAULT 5 CHECK(priority >= 1 AND priority <= 9),
    keywords TEXT NOT NULL DEFAULT '[]',
    source TEXT DEFAULT 'conversation',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

| Column | Purpose |
|---|---|
| `id` | Auto-increment primary key |
| `key` | Unique identifier (e.g. `prefers_go`, `transport_stdio`) |
| `value` | The actual memory content |
| `project` | Namespace this memory belongs to (e.g. `locolm`, `user`, `project-x`) |
| `priority` | 1=critical (always recall), 5=normal, 9=niche (search only) |
| `keywords` | JSON array of searchable terms |
| `source` | How it was created: `conversation` (user told you), `tool` (extracted by a tool), `system` (generated by locolm itself) |
| `created_at` / `updated_at` | Timestamps |

**Project** is a free-form string, not an enum. The LLM names projects naturally: "save this to project locolm", "what's in project user-preferences?" Multiple projects coexist. `general` is the default for uncategorized memories.

### memory_links table (the graph layer)

```sql
CREATE TABLE IF NOT EXISTS memory_links (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    from_id INTEGER NOT NULL REFERENCES memories(id) ON DELETE CASCADE,
    to_id INTEGER NOT NULL REFERENCES memories(id) ON DELETE CASCADE,
    relation TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(from_id, to_id, relation)
);
```

| Column | Purpose |
|---|---|
| `from_id` | Source memory node |
| `to_id` | Target memory node |
| `relation` | Type of relationship |

This is the **triplet model**: `(memory_A, relation, memory_B)` — a directed graph where nodes are memories and edges are typed relationships.

Example data:
```
(12, 15, "supports")     — "user prefers stdio" supports "transport is stdio"
(15, 18, "part_of")      — "transport is stdio" is part of "locolm architecture"
(20, 12, "contradicts")  — "user wants HTTP transport" contradicts "user prefers stdio"
```

**Common relation types:**
- `supports` — A reinforces B
- `contradicts` — A conflicts with B
- `related` — A and B are generally related
- `derived_from` — A was learned from B
- `part_of` — A is a component/attribute of B
- `depends_on` — A requires B

### FTS5 virtual table

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS memories_fts USING fts5(
    key,
    value,
    keywords,
    content='memories',
    content_rowid='id'
);
```

### Indexes

```sql
CREATE INDEX IF NOT EXISTS idx_memories_project ON memories(project);
CREATE INDEX IF NOT EXISTS idx_memories_priority ON memories(priority);
CREATE INDEX IF NOT EXISTS idx_memories_updated ON memories(updated_at DESC);
CREATE INDEX IF NOT EXISTS idx_links_from ON memory_links(from_id);
CREATE INDEX IF NOT EXISTS idx_links_to ON memory_links(to_id);
CREATE INDEX IF NOT EXISTS idx_links_relation ON memory_links(relation);
```

### FTS sync triggers

```sql
CREATE TRIGGER IF NOT EXISTS memories_ai AFTER INSERT ON memories BEGIN
    INSERT INTO memories_fts(rowid, key, value, keywords)
    VALUES (new.id, new.key, new.value, new.keywords);
END;

CREATE TRIGGER IF NOT EXISTS memories_ad AFTER DELETE ON memories BEGIN
    INSERT INTO memories_fts(memories_fts, rowid, key, value, keywords)
    VALUES ('delete', old.id, old.key, old.value, old.keywords);
END;

CREATE TRIGGER IF NOT EXISTS memories_au AFTER UPDATE ON memories BEGIN
    INSERT INTO memories_fts(memories_fts, rowid, key, value, keywords)
    VALUES ('delete', old.id, old.key, old.value, old.keywords);
    INSERT INTO memories_fts(rowid, key, value, keywords)
    VALUES (new.id, new.key, new.value, new.keywords);
END;
```

## Tool Definitions

### memory_recall

Called automatically at conversation start. Loads high-priority memories.

**Input:**
```json
{
  "project": "user",
  "limit": 10
}
```

| Parameter | Required | Default | Description |
|---|---|---|---|
| `project` | no | all | Filter to a specific project |
| `limit` | no | 10 | Max memories to return (hard cap: 50) |

**Behavior:**
- Query memories: `SELECT id, key, value, project, priority, created_at FROM memories WHERE priority <= 3 [AND project = ?] ORDER BY priority ASC, updated_at DESC LIMIT ?`
- For each recalled memory, also query its links: `SELECT ml.relation, m.key, m.project FROM memory_links ml JOIN memories m ON ml.to_id = m.id WHERE ml.from_id = ?`
- Returns compact text block for injection into LLM context
- If `project` is specified, only that project's high-priority memories
- If no `project`, returns high-priority memories across all projects
- `limit` is clamped to max 50 to prevent context overflow

**Response format:**
```
High-priority memories:

[project: user] prefers_go — User prefers Go for systems programming (priority: 1, saved 2026-06-20)
  → supports → transport_stdio
[project: user] prefers_tabs — User prefers tabs over spaces (priority: 2, saved 2026-06-15)
[project: locolm] mcp_version — MCP protocol version is 2024-11-05 (priority: 2, saved 2026-06-10)
  ← derived_from ← mcp_config
```

Recalled memories include their outgoing (`→`) and incoming (`←`) links so the LLM has graph context at conversation start.

### memory_search

On-demand keyword search across memories.

**Input:**
```json
{
  "query": "MCP transport",
  "project": "locolm",
  "limit": 5
}
```

| Parameter | Required | Default | Description |
|---|---|---|---|
| `query` | yes | — | Keywords to search for |
| `project` | no | all | Filter to a specific project |
| `limit` | no | 5 | Max results |

**Behavior:**
- Tokenize `query` on spaces. For each term: wrap in double-quotes (escaping internal quotes), append `*` for prefix matching. Join terms with ` OR `.
- Example: query `"MCP transport"` becomes MATCH expression: `"MCP"* OR "transport"*`
- Query: `SELECT m.key, m.value, m.project, m.priority FROM memories_fts f JOIN memories m ON m.id = f.rowid WHERE memories_fts MATCH ? [AND m.project = ?] ORDER BY rank LIMIT ?`
- Results ordered by FTS5 rank (relevance)

**Behavior (links):**
- For each search result, also query its links: `SELECT ml.relation, m.key, m.project FROM memory_links ml JOIN memories m ON ml.to_id = m.id WHERE ml.from_id = UNION ALL SELECT ml.relation, m.key, m.project FROM memory_links ml JOIN memories m ON ml.from_id = m.id WHERE ml.to_id = ?`
- Links are included inline below each result

**Response format:**
```
Found 3 memories matching "MCP transport":

[project: locolm] mcp_version — MCP protocol version is 2024-11-05 (priority: 2)
  ← derived_from ← mcp_config
[project: locolm] transport_pref — User prefers stdio transport for local tools (priority: 1)
  → supports → mcp_config
[project: general] mcp_config — MCP server listens on 127.0.0.1:11501 (priority: 3)
  → part_of → mcp_server
```

**Note:** If `query` is empty or contains only stop words after tokenization, return an error "query must contain at least one search term".

### memory_save

Saves a new memory or updates an existing one.

**Input:**
```json
{
  "key": "transport_stdio",
  "value": "User prefers stdio transport for local MCP tools",
  "project": "locolm",
  "priority": 2,
  "keywords": ["stdio", "transport", "local", "tools"]
}
```

| Parameter | Required | Default |
|---|---|---|
| `key` | yes | — |
| `value` | yes | — |
| `project` | no | `general` |
| `priority` | no | 5 |
| `keywords` | no | auto-extracted from key + value |
| `source` | no | `conversation` |

**Behavior:**
- If `keywords` not provided: auto-extract from key and value (see Keyword Auto-Extraction section below)
- Upsert on `key` conflict
- Updates `updated_at` timestamp
- FTS triggers handle index updates
- **After saving, the LLM should check for links** — see Memory Linking section in the System Prompt

**On update (key already exists):**
- Replace `value`, update `updated_at`
- Re-extract keywords if `keywords` not provided
- The LLM should re-evaluate links — the new value may create or invalidate relationships
- Do NOT change `project` or `priority` on update unless explicitly provided (preserve user's original categorization)

### memory_get

Retrieves a single memory by key, including its links.

**Input:**
```json
{
  "key": "transport_stdio"
}
```

**Behavior:**
- Query: `SELECT id, key, value, project, priority, keywords, source, created_at, updated_at FROM memories WHERE key = ?`
- Also query links (both directions)
- Returns full memory with all links, or "not found"

**Response format:**
```
[project: locolm] transport_stdio — User prefers stdio transport for local MCP tools (priority: 2, saved 2026-06-20)

Links:
  → supports → mcp_config
  ← part_of ← transport_layer
```

### memory_forget

Deletes a memory by key and all its links.

**Input:**
```json
{
  "key": "transport_stdio"
}
```

**Behavior:**
- `DELETE FROM memories WHERE key = ?`
- Cascading delete on `memory_links` rows where `from_id` or `to_id` matches (via FK ON DELETE CASCADE)
- FTS trigger handles index cleanup
- Returns confirmation or "not found" message

**Warning:** This also destroys all links to/from this memory. The LLM should confirm with the user before forgetting a memory that has links (check with `memory_links` first if unsure).

### memory_link

Creates a typed relationship between two memories.

**Input:**
```json
{
  "from_key": "transport_stdio",
  "to_key": "mcp_config",
  "relation": "part_of"
}
```

| Parameter | Required | Default |
|---|---|---|
| `from_key` | yes | — |
| `to_key` | yes | — |
| `relation` | no | `related` |

**Behavior:**
- Look up both memories by key
- Validate `relation` is one of: `supports`, `contradicts`, `related`, `derived_from`, `part_of`, `depends_on`. If not, return error listing valid types.
- Insert into `memory_links` (ignore if duplicate via UNIQUE constraint)
- Returns confirmation or error if either key not found

### memory_links

Retrieves linked memories for a given key.

**Input:**
```json
{
  "key": "transport_stdio",
  "direction": "both"
}
```

| Parameter | Required | Default | Description |
|---|---|---|---|
| `key` | yes | — | Memory key to find links for |
| `direction` | no | `both` | `out`, `in`, or `both` |

**Behavior:**
- Look up memory by key
- Query `memory_links` for rows where `from_id` or `to_id` matches
- Return linked memories with their relation type

**Response format:**
```
Linked memories for "transport_stdio":

→ part_of → mcp_config: MCP server listens on 127.0.0.1:11501
← supports ← prefers_stdio: User prefers stdio for local tools
```

Arrow legend: `→ relation →` means "this memory points to" (from_id = this). `← relation ←` means "points to this memory" (to_id = this).

### memory_stats

Returns an overview of the memory system.

**Input:**
```json
{}
```

**Behavior:**
- Query counts: `SELECT project, COUNT(*) FROM memories GROUP BY project`
- Query priority distribution: `SELECT priority, COUNT(*) FROM memories GROUP BY priority`
- Query total links: `SELECT COUNT(*) FROM memory_links`
- Returns a summary overview

**Response format:**
```
Memory system overview:

Total memories: 47
Total links: 23

By project:
  user: 12
  locolm: 28
  general: 7

By priority:
  1-3 (critical): 5
  4-6 (normal): 30
  7-9 (niche): 12
```

**Use case:** Helps the LLM understand the current state of the knowledge graph. Useful when deciding whether to search or recall, and for answering user questions about "what do you know?"

### memory_unlink

Removes a typed relationship between two memories.

**Input:**
```json
{
  "from_key": "transport_stdio",
  "to_key": "mcp_config",
  "relation": "part_of"
}
```

| Parameter | Required | Default | Description |
|---|---|---|---|
| `from_key` | yes | — | Source memory key |
| `to_key` | yes | — | Target memory key |
| `relation` | yes | — | Specific relation type to remove |

**Behavior:**
- Look up both memories by key
- Delete matching row from `memory_links`
- Returns confirmation or "no matching link found" message

**Who initiates:** User explicitly asks to remove a link, or the LLM realizes a link is incorrect or stale (see Memory Linking → When NOT to Link in the System Prompt).

## Keyword Auto-Extraction

When `keywords` is not provided in `memory_save`:

1. Split `key` on `_`, `-`, spaces, and camelCase boundaries (e.g. `prefersGo` → `prefers`, `Go`)
2. Split `value` into words, take first 20
3. Lowercase all terms
4. Remove stop words (the, a, an, is, are, of, to, for, in, on, and, or, with, from, by, at, be, this, that, it, as, s, t, ve, ll, re, d)
5. Remove terms shorter than 3 characters
6. Deduplicate
7. Cap at 30 keywords maximum
8. Store as JSON array

The LLM can always provide explicit keywords for better results.

## System Prompt

Replace `prompt.md` with updated instructions for the new memory tools.

The prompt is the **contract** between the locolm environment and any LLM. It defines all behavior, including when and how to link memories.

### Updated prompt.md

```markdown
# locolm — System Prompt

You are a helpful AI assistant running locally through locolm, a local AI platform that gives you access to the web, your own memory, and a locally hosted LLM.

## Available Tools

You have access to the following tools. Use them whenever appropriate to fulfill the user's request.

### Web & Research

- **google_search** — Search the web using Google Custom Search. Use this to find information, news, documentation, or anything not in your memory or training data. Always search when the user asks about something you're unsure about.
- **web_fetch** — Fetch and read the full content of a web page or PDF. Use this after google_search to read specific results, or when the user provides a URL and wants you to read it. Extracts clean article text from HTML pages and plain text from PDF documents.

### Memory & Persistence

- **memory_recall** — Load high-priority memories. **Call this at the start of every conversation.** Optionally filter by project. Returns essential context with links.
- **memory_search** — Search memories by keywords. Use during conversation when you need to find specific information. Can filter by project. Returns matching memories ranked by relevance, with links.
- **memory_save** — Save a new memory or update an existing one. Requires `key` and `value`. Optional: `project`, `priority`, `keywords`, `source`. **After saving, check for links.**
- **memory_get** — Retrieve a single memory by key with its links. Use when you need to inspect a specific memory.
- **memory_forget** — Delete a memory by its key. **Warning:** also destroys all links to/from it. Check links first if unsure.
- **memory_link** — Create a typed relationship between two memories. Requires `from_key`, `to_key`, and `relation`.
- **memory_unlink** — Remove a typed relationship between two memories. Requires `from_key`, `to_key`, and `relation`.
- **memory_links** — View memories linked to a given memory. Shows relationships in both directions.
- **memory_stats** — Overview of the memory system: counts by project, priority distribution, total links.

## Memory Projects

Memories are organized into **projects** (namespaces). When saving a memory, specify which project it belongs to. Examples:
- `user` — Facts and preferences about the user
- `locolm` — Information about the locolm project itself
- `project-x` — Any other named project the user creates

Projects are created implicitly — just use a new project name in `memory_save` and it exists.

## Memory Priorities

- 1-3: Critical — always loaded via memory_recall
- 4-6: Normal — loaded via memory_search when relevant
- 7-9: Niche — only loaded on specific search

## Memory Linking

Memory linking builds a knowledge graph. Links are **directed** and **typed**: memory A relates to memory B in a specific way.

### Relation Types

- `supports` — A reinforces B
- `contradicts` — A conflicts with B
- `related` — A and B are generally related
- `derived_from` — A was learned from B
- `part_of` — A is a component/attribute of B
- `depends_on` — A requires B

### Who Initiates Linking

**Both you and the user can initiate linking and unlinking.**

**User-initiated linking:** The user may explicitly instruct you to link two memories. When the user says things like "link X to Y" or "connect A with B", call `memory_link` immediately with the appropriate relation type.

**LLM-initiated linking (automatic):** You should proactively build the knowledge graph by creating links when you recognize relationships. This happens in three situations:

1. **During memory_save:** After saving a new memory, check if it relates to any memories you currently have in context (from recall or recent search results). If so, call `memory_link` to create the connection. You do NOT need to ask the user for permission — just do it.

2. **During memory_save (update):** When updating an existing memory, re-evaluate its links. The new value may create new relationships or invalidate existing ones. Call `memory_link` for new connections and `memory_unlink` for broken ones.

3. **During conversation:** When discussing a topic that connects two or more memories you already know about, create the link. For example, if the user asks about MCP transport and you recall both "transport_stdio" and "mcp_config", link them automatically.

**User-initiated unlinking:** The user may say "unlink X from Y" or "remove the link between A and B". Call `memory_unlink` immediately.

**LLM-initiated unlinking:** You should remove links when you realize they are incorrect or stale. Situations:
- A `supports` link where the source memory has been updated and no longer supports the target
- A `related` link that turned out to be wrong
- A `derived_from` link where the derivation is no longer accurate

### When NOT to Link

- Don't link trivial or obvious relationships that add no value
- Don't create redundant links (if A→B "related" exists, don't add A→B "related" again)
- Don't link memories from different projects unless the relationship is meaningful
- Don't force links — if no clear relation type fits, don't link

### Complete Memory Lifecycle

1. **Create** — `memory_save` with a new key. Auto-extract keywords. Then check for links against memories in current context.
2. **Retrieve** — `memory_recall` at conversation start (includes links). `memory_search` during conversation (includes links). `memory_get` for a single memory.
3. **Update** — `memory_save` with existing key. Re-extract keywords. Re-evaluate links (add new, remove stale).
4. **Link** — `memory_link` creates a directed typed edge. Initiated by user instruction or LLM recognition.
5. **Unlink** — `memory_unlink` removes a specific edge. Initiated by user instruction or LLM recognition of incorrect/stale links.
6. **Delete** — `memory_forget` removes the memory and cascades all links. Check `memory_links` first if the memory has connections.
7. **Discover** — Links are surfaced in `memory_recall`, `memory_search`, and `memory_get`. Use `memory_links` to explore the graph. Use `memory_stats` for an overview.

### Discovering Links

- `memory_recall` results include links to/from each recalled memory, so you see the graph at conversation start.
- `memory_search` results include links alongside search matches.
- `memory_get` returns the memory with all its links.
- Use `memory_links` to explore what's connected to a memory you just found.
- Use `memory_unlink` to remove incorrect or outdated links.

## Tool Usage Guidelines

1. **Always call memory_recall at the start of a new conversation** to load essential context and the link graph.
2. **Use memory_search during conversation** when the discussion might have related memories. Extract keywords from what the user is saying.
3. **Use memory_get** when you need to inspect a specific memory and its links (e.g., before forgetting).
4. **After every memory_save, check for links** — does this new memory relate to anything in your current context? If yes, link them. On update, also re-evaluate existing links.
5. **Use memory_link** when two memories have a relationship — whether the user asked or you recognized it yourself.
6. **Use memory_unlink** to remove incorrect or stale links (user asks, or you realize a link is wrong).
7. **Use memory_links** to explore what's connected to a memory you just found.
8. **Use memory_stats** to understand the overall state of your knowledge graph.
9. **Use google_search when you need current information** — your training data may be outdated.
10. **Use web_fetch to read specific pages** — search results only show snippets.
11. **Save important information with memory_save** — if the user tells you something important, save it.
12. **Be efficient** — don't search for things you already know from recall, and don't save trivial or temporary information.

## Behavior

- Be concise and direct. Don't repeat information the user already knows.
- If you don't know something, search for it rather than guessing.
- If the user corrects you, save the correction to memory.
- Prefer using memory to maintain context across conversations rather than asking the user to repeat yourself.
- Build the knowledge graph proactively — it makes you smarter over time.
```

```sql
-- 4. Verify, then drop legacy table
```
